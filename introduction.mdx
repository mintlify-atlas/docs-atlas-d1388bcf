# Example Voting App

The Example Voting App is a reference implementation of a distributed, multi-container application. It demonstrates how real-world workloads can be composed from multiple independent services — each written in a different language — and orchestrated together using Docker tools.

## What it does

The app presents users with a simple voting interface where they can choose between two options (default: **Cats** vs **Dogs**). Votes are queued through Redis, processed by a background worker, persisted to PostgreSQL, and surfaced in real time on a results dashboard.

## Stack overview

| Service | Technology | Purpose |
|---------|-----------|---------|
| `vote` | Python / Flask | Front-end voting interface |
| `result` | Node.js / Express | Real-time results dashboard |
| `worker` | .NET / C# | Async vote processor |
| `redis` | Redis | Vote message queue |
| `db` | PostgreSQL | Persistent vote storage |

## Deployment targets

The same application can be deployed three ways without changing application code:

- **Docker Compose** — single-node local development
- **Docker Swarm** — multi-replica clustered deployment
- **Kubernetes** — cloud-native orchestration via YAML manifests

## What this app demonstrates

<AccordionGroup>
  <Accordion title="Polyglot microservices">
    Each service is written in a different language (Python, JavaScript, C#), showing how Docker lets you pick the best tool for each job without coupling your stack.
  </Accordion>
  <Accordion title="Asynchronous messaging">
    The vote service writes to a Redis list. The worker pops from that list independently, decoupling vote submission from vote processing.
  </Accordion>
  <Accordion title="Real-time data push">
    The result service queries PostgreSQL on a 1-second polling loop and broadcasts updated scores to all connected browsers via Socket.IO WebSockets — no page refresh required.
  </Accordion>
  <Accordion title="Health-checked dependencies">
    Every service that depends on Redis or PostgreSQL uses Docker health checks to wait for those dependencies to be ready before starting, preventing race conditions on startup.
  </Accordion>
  <Accordion title="Multi-stage Docker builds">
    The vote service Dockerfile uses multi-stage builds with separate `dev` and `final` stages, enabling fast local iteration with watchdog hot-reload while producing a lean production image with gunicorn.
  </Accordion>
</AccordionGroup>

<Note>
  This app is intentionally simple. It is not a production-ready template — it is a learning resource for understanding how Docker orchestrates multi-service applications.
</Note>
